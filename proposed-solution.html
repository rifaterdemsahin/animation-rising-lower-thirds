🤖 Proposed Solution: One Agent Request, Not Five Manual Steps

A proposal, not something built yet — this page describes what could be built next, not what exists today. See mcp-guide.html for what's actually live right now, and cost-analysis.html for the time/cost this would collapse.

❗ The problem this solves

Going from "I have an idea for a lower-third animation" to a finished asset currently touches several separate manual steps, per cost-analysis.html: generate a background image (Gemini/fal.ai, by hand), work out placement/timing (the Builder, by hand), then render or export (a form on this site, or Canva, by hand). Each step works well on its own, but nothing chains them together — a person has to be in the loop at every handoff.

💡 The proposal

Extend the already-deployed Go API (server/, live at lower-thirds-api.fly.dev) with a few more capabilities, and expose all of them as MCP tools — so an agent can go from a plain-English request to a finished video in one turn, calling the pipeline itself instead of a person clicking through four different pages.

Proposed MCP toolDoesBuilds on
generate_background(prompt) Calls fal.ai's nano-banana-2 endpoint server-side, returns/stores the generated image The image-generation skill's approach, moved server-side instead of a local one-off script
generate_html(cards, background) Renders an explainer.html-style page for the given content, for when the deliverable is an interactive page, not a video assets/js/lower-thirds.js — the same engine, driven server-side
render_mixed_video(cards, background, timing) The "mixed video" — composites the (generated or supplied) background with the full animated card set into one exported video file, server-side The existing /render endpoint's chromedp + MediaRecorder approach — extended to N cards and an optional background, closing the gap MASTER_SPEC.md §8.4 flags today

All three (plus the existing /render) wrapped as proper MCP tools — not just plain REST endpoints — per MCP_GUIDE.md's "Option 2/3" pattern, just with more of the pipeline on the server side instead of the client/agent side.

🔄 What the request would look like

  1. You ask an agent: "Make me a lower-thirds video explaining an auth flow: client → auth service → database."
  2. Agent calls generate_background with a content-matched prompt (following the standing rule in MASTER_SPEC.md §10.2 — the scene layout mirrors the actual flow, not a generic backdrop).
  3. Agent calls render_mixed_video with that background plus the three stages and their timing.
  4. Agent hands back the finished video — no manual prompting, no manual Builder placement, no manual Canva work.

⚠️ What this would actually take

  • Cost control. Server-side image generation means the API itself starts spending real money per call (see cost-analysis.html — ~$0.08/image). Needs either callers supplying their own FAL_KEY, or a hard quota on a shared one, before this is safe to expose broadly.
  • Latency. Chaining image generation + headless Chrome + video capture in one request could easily exceed a reasonable HTTP timeout (the current /render already uses a 45s context budget). This would likely need to become an async job — submit, poll, download — rather than one synchronous call.
  • Auth. The current single shared API_TOKEN doesn't distinguish callers or track usage — fine when the only cost is compute, less fine once each call can cost real money.

None of this is started — server/main.go still only implements the 2-card /render described in mcp-guide.html today.
