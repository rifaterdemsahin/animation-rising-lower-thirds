🎬 Rising Lower Thirds

A practical animation helper for lower-third cards that spring up from off-screen, bounce, and settle into place, each one staggered behind the last. The formal motion spec was designed against the twin "Real Jobs / Real People" example (two cards, 150ms stagger) — the full math lives in problem.md, and this particular rising-card animation style is demonstrated in this video. The interactive tools below default to a different worked example — a three-stage Kubernetes request flow — built on that same spec.

🌐 Domain

Broadcast/live-stream motion graphics — specifically lower thirds, the text+icon bars overlaid near the bottom of a video frame to identify a speaker, role, or topic. This sits at the intersection of video editing (a reusable graphic for a recorded timeline), live production (a real-time browser-source overlay for streaming), and front-end animation (the spring-motion implementation itself, in CSS/Canvas).

🧩 The problem

problem.md is the formal design/motion spec this project implements: two rounded rectangle cards (dark teal, 85% opacity, 24–30px radius), a cyan icon on the left and serif text on the right, that rise from off-screen and settle with a damped-spring bounce — the second card entering 150ms after the first. It also defines the reference screenshot below and the exact cubic-bezier/damped-harmonic math the animation should follow.

✅ Requirements

  • Type the text, get the animation — no manual keyframing per project.
  • Visual spec compliance — dual rounded rectangles (24–30px radius, #0B1B1E @ 85% opacity), cyan icon + serif text, 24px gap, per problem.md.
  • Motion spec compliance — damped harmonic rise (ζ=0.5), opacity to 100% by frame 8, card 2 staggered +150ms behind card 1.
  • Three output modes from one shared animation engine: an actual downloadable video file with a background (for editing), a transparent live overlay with no background (for OBS/vMix streaming), and an interactive HTML teaching page with a background — all defaulting to the same Kubernetes example content.
  • Usable three ways — click through the UI, drive it headlessly via a URL query-param API or a plain JS API, or call the hosted Go/Fly.io API (see MCP_GUIDE.md / mcp-guide.html).
  • No build step — plain static HTML/CSS/JS, deployable as-is to GitHub Pages.

💡 What we're trying to do

Instead of hand-keyframing this animation in a video editor every time the copy changes, you type the text you want for each lower third and the same animation engine (assets/js/lower-thirds.js) renders it for you. One engine, one shared example (LowerThirds.PRESETS.kubernetes: Cluster → Ingress Gateway → Pod), three outputs:

  • video-recorded.html — type your text, render it to a canvas with the Kubernetes background painted behind the cards, and export an actual downloadable video file (WebM/MP4) you can drop into an edit timeline.
  • video-live.html — the same animation as a lightweight, transparent-background overlay you can point OBS/vMix at for a live stream, driven by URL query params or the JS API directly — no background image here, since a live overlay has to stay transparent to composite over your stream.
  • explainer.html — the same three cards, staged one at a time instead of together, with the Kubernetes background, flow arrows, and a Play button that explains the concept. Has a "Download standalone HTML" button too.

This is a small toolbox, not just two forms: every page works by clicking through the UI, but every page is also drivable without touching the UI at all — a URL query-param API for scripting/automation, and a plain JS API (LowerThirds / LowerThirdsRecorder) for embedding the engine in your own page. Each page documents both under an "API"/"Reusable" panel.

See recommendation.html for suggested next steps, informed by the video-production tooling evaluated for this project, MCP_GUIDE.md for driving this toolbox from an MCP client/agent instead of the UI, and master-spec.html for the full design/implementation spec behind the Kubernetes example and the shared-background/preset work.

🔬 Same animation, three different environments

All three outputs start from the same input (the same three icon/text fields, the same Kubernetes example by default) and run through the same motion math — springState() in assets/js/lower-thirds.js never changes between them. What comes out the other side looks similar at a glance, but each environment has a genuinely different job, so each one needs different things to do that job — that's why they're not just the same page with a different button.

🎥 Recorded Video📡 Live Overlay🧭 Explainer
Input Typed icon/text × 3 (Kubernetes default) Same Same, plus per-stage timing (0/3200/6400ms instead of together)
Rendering path Canvas 2D — drawCanvasFrame() DOM/CSS — LowerThirds.mount() DOM/CSS mount() + a background layer, SVG flow arrows, and captions
Output A downloadable video file (WebM/MP4) — a portable artifact A live, real-time transparent layer — not a file, a feed An interactive page a person clicks through themselves
Background Baked in — painted onto the canvas every frame None, deliberately — must stay transparent Yes — content-matched to the labels, for standalone context
What it needs that the others don't canvas.captureStream() + MediaRecorder; everything has to be pixel-baked because the output file has to be self-contained after the browser closes Genuine transparency, so OBS/vMix can composite it over whatever video is already playing underneath Captions, arrows, and the Builder's parameter/position tools — its job is explaining a concept, not feeding another system

In short: the animation is one shared thing: three outputs because three different downstream jobs — editing into a timeline, compositing live, or teaching someone standing alone — each impose their own requirements on how that same motion has to be packaged.
