💰 Cost Analysis

What each environment actually needs as input, what it produces, roughly how long it takes to build (one-time) versus use (per project), what a later change costs (often very different from the first build), what a script changing everything at once costs at volume, and the only two places real money changes hands in this whole system. See index.html's "Same animation, three different environments" section for how the outputs differ visually; this page is about effort and cost specifically.

🔄 The cost of change — what happens the second time

The table above is first creation. Real projects don't stop there — text gets revised, a stage gets renamed, someone asks for a different color. What that costs depends entirely on whether the output is baked (a finished file — changing it means redoing the export) or parametric (rendered live from inputs — changing it just means changing the input, at the same cost as the first time, not a fresh "editing" tax on top).

📡 Live Overlay Parametric — nothing is ever "finished," it's rendered fresh each time it loads Instant — edit the form/URL params, it's live; there's no separate old artifact to invalidate N/A — never has a background

The pattern: anything parametric (Live Overlay, Explainer's page, a live Canva design, the API) costs the same to change as it did to create the first time — a change isn't a special, more expensive operation, it's just running the same thing again with different input. The only genuinely recurring cost is regenerating a baked image (a background) or a baked export (a rendered video file, or a downloaded Canva export) — and even that's the same ~$0.08/regeneration whether it's the first time or the fiftieth.

🤖 Scripted generation — when the background and the text/emojis both change every time

The section above assumes one thing changes at a time (just text, or just the background). A script or an MCP agent driving this programmatically (the URL API, the JS API, or the hosted API — see mcp-guide.html) is a different case: it's realistic for every field — icons, text, and the background prompt — to be different on every single call, e.g. generating a batch of N videos that each explain a different flow.

  • Cost per unique asset ≈ $0.08, not per field changed. The text/emoji fields are parametric — however many of them differ from the last run, they add $0. The only thing that costs money is the background image, so cost scales with how many distinct backgrounds you generate, not how many labels or how much text you vary. A batch of 50 videos with 50 different topics and 50 different backgrounds ≈ 50 × $0.08 ≈ $4. The same 50 videos sharing one background ≈ $0.08 total.
  • It's two calls today, not one. Nothing in this repo currently lets a script generate the background and the animation in a single request — server/main.go's /render only builds the animation from a background you already have. A script has to (1) generate the image itself (e.g. via the same image-generation skill/fal.ai call this project uses) and (2) separately call this project's URL/JS/ hosted API with the result. proposed-solution.html describes what collapsing that into one call would take.
  • Batch-friendly mitigation: if many generated assets share a visual theme (e.g. "explain a Kubernetes flow" reused across a whole batch with only the labels changing), generate one background and pass its path/URL into every call in the batch instead of a new prompt each time — same tactic as canva-value.html's reusable image library, just applied to a scripted loop instead of manual Canva duplication.

💵 Where money actually gets spent

Everything above is free static hosting and browser-side computation except two things:

  1. Image generation (Gemini "Nano Banana" via fal.ai's nano-banana-2 endpoint, used for backgrounds). Priced per output megapixel, roughly $0.0398/MP — this project's 1920×1080 background is ~2.07MP, so about $0.08 per generation. You only pay when you actually generate a new image; reusing assets/k8s-explainer-background.png (or any background you've already generated) costs nothing.
  2. Fly.io hosting for the API in server/ (lower-thirds-api.fly.dev). The machine is configured to auto-stop when idle and auto-start on request (auto_stop_machines/min_machines_running: 0 in server/fly.toml), so cost scales with actual usage rather than running 24/7 — exact numbers depend on your Fly.io plan and request volume, which this page can't state precisely without checking live billing; check the Fly.io dashboard for your account's current usage.

Everything else — the static site (GitHub Pages), the animation engine, the Builder tools, the standalone HTML export, and driving any of it via the URL API or a browser-automation MCP client — costs nothing to run, no matter how many times you use it.

🤔 Which is cheapest for your situation

  • Reusing existing content (same background, new labels) → free, any environment, seconds.
  • New content, need a video file → 🎥 Recorded Video + a new background (~$0.08, once).
  • New content, need it live → 📡 Live Overlay (no background needed at all — it must stay transparent).
  • New content, need to explain/teach it → 🧭 Explainer + a new content-matched background (~$0.08).
  • No code, want to hand-tweak in Canva → 🎨 the manual path — most generation cost (~$0.30+) and most time (~15–30 min) the first time; see canva-value.html for why that's still often worth it and how to cut it down on every project after the first.

See proposed-solution.html for a proposal that would collapse most of this into a single agent request.
