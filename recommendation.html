Recommendations

Before building the video-production pieces of this project, I surveyed the current landscape of Claude Code "video production" skills on GitHub (ychoi-kr/claude-ffmpeg-skill, digitalsamba/claude-code-video-toolkit, wilwaldon/Claude-Code-Video-Toolkit). Here's what's in the project now and what I'd do next.

What's already installed

.claude/skills/ffmpeg-usage — an MIT-licensed ffmpeg skill (from ychoi-kr/claude-ffmpeg-skill) for format conversion, compression, GIF creation, and platform-specific presets. This is the one piece that's genuinely useful for this repo today: video-recorded.html exports WebM via MediaRecorder, and any Claude Code session in this repo can now call on that skill to convert/compress the output (e.g. to MP4 for an editor, or a small GIF for a README).

What I deliberately didn't add

Toolkits like digitalsamba/claude-code-video-toolkit and wilwaldon/Claude-Code-Video-Toolkit bundle Remotion/Manim pipelines, screen-recording, and multi-stage render workflows. Those are built for producing longer explainer/promo videos from a Node/React toolchain. This repo is a static site deployed as-is via GitHub Pages (.github/workflows/static.yml, no build step) — pulling in a Node/Remotion dependency would change that deployment model for a feature we don't need yet (we only need one two-card lower-third clip, not a multi-scene video). If that changes, Remotion is the right upgrade path — see below.

Recommended next steps

  1. Use the ffmpeg skill to normalize output. Browser-recorded WebM varies by browser/codec support. Standardize on one delivery format (e.g. H.264 MP4 at 1920×1080) via the skill's recipes rather than shipping whatever the browser happened to record.
  2. Move to Remotion only if requirements grow — e.g. you need frame-exact reproducibility (MediaRecorder is real-time and can drop frames under load), audio sync, or to batch-render many variants server-side/in CI. Remotion's spring() helper can reproduce the same damped harmonic motion already implemented in assets/js/lower-thirds.js, so the math wouldn't need to change, only the renderer.
  3. Sync timing against real footage. The 150ms stagger and spring constants (ζ = 0.5, ωn = 11) in problem.md were tuned by eye — validate them against actual voiceover/cut points once this is used in a real edit, and adjust STAGGER_MS / ZETA / WN in lower-thirds.js if the bounce reads too slow or too snappy.
  4. Batch generation. If you end up needing many lower-thirds in one project (multiple names/roles), drive video-recorded.html from a small JSON/CSV list instead of typing each one by hand, then use the ffmpeg skill to concatenate the individual clips.
  5. Keep video-live.html as the source of truth for "does it look right live" — it's the cheapest way to sanity-check timing/wording against a real stream layout before committing to a recorded export.
