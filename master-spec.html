🗂️ Master Spec: Kubernetes Flow Explainer

The design/implementation spec for explainer.html and the engine generalization that made it possible — plus (§8) how that Kubernetes example became the shared default across all three outputs, and (§9) a full audit-style reference to every other doc in this repo. This page mirrors MASTER_SPEC.md on-site. It sits alongside problem.md (the unmodified motion/visual contract for the base 2-card spring) and does not change it.

Status: ✅ implemented (2026-07-17), 🔄 extended (2026-07-18, see §8–9 below). Scope decided via clarifying questions on 2026-07-17: new standalone page, background image pre-generated as a static asset, sequential staged reveal (Cluster → Ingress Gateway → Pod). Extended 2026-07-18: the Kubernetes example became the shared default content across all three outputs, not just explainer.html — §8 covers what changed and why, §9 is a full project reference.

🎯 1. Goal

Before this spec, the repo had one shared animation engine (assets/js/lower-thirds.js) driving two outputs, both built around the same fixed "Real Jobs / Real People" two-card example:

  • 🎥 video-recorded.html → downloadable MP4/WebM
  • 📡 video-live.html → transparent live overlay (OBS/vMix)

This spec adds a third output and a worked Kubernetes example:

  • 🧭 explainer.html (new) → an interactive HTML page that uses the engine to visually teach one concept — how a request flows through a Kubernetes test cluster — via three staged lower-third cards, a generated background, and a single Play button.

The K8s content is the flagship demo for the new HTML output specifically. The engine change that makes it possible (support for N cards, not just 2) is generic and available to all three outputs.

⚙️ 2. Engine change: generalize 2 cards → N cards

lower-thirds.js hardcoded config.card1 / config.card2 in both mount() and drawCanvasFrame(). springState(t, delayMs) already took an arbitrary delay, so it already supported N cards — only the renderers needed to generalize.

Change: accept config.cards = [{icon, text, delayMs}, ...] (array, arbitrary length) in both renderers, laid out left-to-right in array order. config.card1/config.card2 keep working unchanged for backward compatibility — internally normalized to cards: [card1, card2] with delayMs 0 and STAGGER_MS respectively, so video-recorded.html and video-live.html needed zero changes.

Each card entry's delayMs is caller-supplied (not a fixed 150ms), so the explainer page can space its three stages seconds apart instead of milliseconds apart. This is the only change to lower-thirds.js — STYLE, springState(), and the spring constants (ζ=0.5, ωn=11, amplitude 120px) are untouched.

🧭 3. New page: explainer.html

3.1 Content — the three stages

StageCarddelayMsCaption
1 🖥️ Kubernetes Test Cluster 0 "Everything below lives inside one test cluster — the boundary for all the resources in this demo."
2 🚪 Ingress Gateway 3200 "External traffic enters through the Ingress Gateway, which matches incoming requests against routing rules."
3 📦 Pod 6400 "The Gateway forwards the matched request to the target Pod running the application."

Dwell timing: each card settles per springState (~1000ms) and then holds for ~2200ms so its caption is readable before the next stage fires — hence the ~3200ms spacing. Starting values to tune by eye, not a hard spec. Card 1 stays visible/pinned as the persistent context label while cards 2 and 3 stage in — it does not exit.

3.2 Flow direction (arrows)

Between stage 1→2 and stage 2→3, a directional connector (SVG line + arrowhead, cyan #00D2D2) is drawn once both endpoint cards are visible. This is DOM/SVG only, layered in explainer.html itself — not a lower-thirds.js engine feature. MP4 export and live overlay stay arrow-free (see §5, out of scope).

3.3 Interactive Play button

  • One ▶ Play button starts the staged sequence from t=0.
  • Becomes ⟳ Replay once the sequence completes.
  • No auto-play on load — but scriptable via ?autoplay=1.

3.4 Background image

  • Generated once, ahead of time, and committed as a static asset — no live API calls from the page.
  • Generated via the image-generation skill (fal.ai nano-banana-2, fal's hosted alias for Gemini 2.5 Flash Image / "Nano Banana").
  • Output location: assets/k8s-explainer-background.png.
  • Used as a full-bleed background on the stage, dimmed via a CSS gradient overlay so card contrast still reads.

3.5 Not a form

Unlike video-recorded.html/video-live.html, this page ships fixed K8s content by default — it's a worked example, not a general-purpose tool (though it now also accepts URL-param overrides, see explainer.html's own "Reusable as a URL API" panel).

🔄 8. Addendum (2026-07-18): unified Kubernetes example across all three outputs

This reverses §5's original "MP4/live-overlay K8s variants: out of scope" call. Decided via clarifying question that day: video-recorded.html and explainer.html both get the Kubernetes background + content; video-live.html switches its default content to the same Kubernetes example but stays transparent (no background image) — a live-compositing OBS/vMix overlay can't carry a baked-in background without losing the ability to composite over the stream underneath it, so that page is deliberately excluded from "all outputs get a background."

8.1 Engine changes

  • LowerThirds.PRESETS.kubernetes — the shared example content (cards, same three entries/delays as §3.1) plus background (the image path), added to lower-thirds.js as the single source of truth. All three output pages read from this preset for their defaults instead of each hardcoding their own copy — avoids the "Real Jobs / Real People" hardcoded-in-three-places drift this same repo had before this addendum.
  • drawCanvasFrame(ctx, w, h, t, config, restY, onBackground) — new optional 6th parameter. Runs right after the per-frame clearRect and before any card is drawn, so a caller can paint a background image without the engine needing to know anything about image loading/decoding. Purely additive.

8.2 video-recorded.html

  • Form grew a third icon/text pair; all three default to the Kubernetes preset's values (typing over them still works exactly as before).
  • A preloaded background image is drawn full-canvas, then the same dark gradient explainer.html uses, via the new onBackground callback.
  • Bug fixed in passing: the recorder's animation loop previously compared elapsed time against the fixed LowerThirds.DURATION_MS (sized for the legacy 2-card case, ~1350ms) regardless of the actual config — would have silently cut a 3-stage Kubernetes recording off after ~1.3s instead of the ~7.6s it needs. Now computes LowerThirds.computeDuration(...) per run.

8.3 video-live.html

  • Same third field added; defaults to the Kubernetes preset's cards; URL API gained icon3/text3.
  • No background image — the one output that intentionally does not follow "all outputs get a background."

8.4 Known gap

The hosted API (server/main.go) still only accepts {icon1, text1, icon2, text2} — not extended to 3 cards or a background as part of this addendum. Calling it still produces a 2-card, background-free video even though the page it drives now defaults to 3 cards with a background when used directly. Documented as a known limitation in mcp-guide.html's Option 3 rather than silently mismatched.
